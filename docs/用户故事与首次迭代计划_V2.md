# 用户故事与首次迭代计划（V2 修订版）

---

## 第一部分：用户故事与验收标准

### 角色定义

| 角色 | 代号 | 说明 |
|------|:----:|------|
| 个人用户 | P | 平台普通使用者，可创建/参与活动、组建小队、添加好友 |
| 商家用户 | M | 需上传执照经审核，可发布活动、维护商家资料 |
| 管理员 | A | 平台运营方，负责用户审核、活动监管、小队管理 |
| 系统 | S | AI 审核、自动通知等自动化能力 |

### Epic 1：用户信息管理（US01-US07）

---

#### US01 个人用户注册

> **作为** 潜在的个人用户（P）
> **我希望** 使用邮箱和密码注册账号
> **以便** 成为平台用户并享受完整功能

**故事点：** 3　|　**优先级：** Must（P0）

| AC# | 场景 | Given（给定） | When（当） | Then（则） |
|:---:|------|-------------|-----------|----------|
| AC1 | 成功注册 | 用户填写未注册的邮箱和符合规则的密码 | 提交注册表单 | 系统发送激活邮件至该邮箱，账户状态为"待激活" |
| AC2 | 邮箱已被注册 | 用户填写的邮箱已被其他账号绑定 | 提交注册表单 | 系统提示"该邮箱已被注册，请更换邮箱或直接登录" |
| AC3 | 密码不合规 | 用户填写的密码长度不足 8 位或缺少字母/数字组合 | 提交注册表单 | 系统提示"密码需至少 8 位，且包含字母和数字"并拒绝提交 |
| AC4 | 邮箱格式错误 | 用户填写的字符串不符合邮箱格式 | 失去焦点或提交 | 系统提示"请输入有效的邮箱地址" |
| AC5 | 激活成功 | 用户收到激活邮件且账户处于"待激活"状态 | 点击邮件中的有效激活链接 | 账户状态变更为"已激活"，跳转至登录页 |
| AC6 | 激活链接过期 | 用户收到激活邮件但链接已超过有效期 | 点击过期链接 | 系统提示"激活链接已过期，请重新发送激活邮件" |

#### US02 个人用户登录

> **作为** 已注册的个人用户（P）
> **我希望** 使用邮箱和密码登录平台
> **以便** 访问平台全部功能

**故事点：** 3　|　**优先级：** Must（P0）

| AC# | 场景 | Given | When | Then |
|:---:|------|------|------|------|
| AC1 | 登录成功 | 用户账户已激活，输入正确的邮箱和密码 | 点击登录 | 系统返回 JWT Token，跳转至首页 |
| AC2 | 密码错误 | 用户输入正确邮箱但密码不匹配 | 点击登录 | 系统提示"邮箱或密码错误"，不清空邮箱字段 |
| AC3 | 账户未激活 | 用户已注册但未点击激活链接 | 点击登录 | 系统提示"账户尚未激活，请先完成邮箱激活" |
| AC4 | 账户已封禁 | 用户账户处于封禁状态 | 点击登录 | 系统提示"账户已被封禁，如有疑问请联系平台" |
| AC5 | 连续失败锁定 | 同一账户连续 5 次输入错误密码 | 第 5 次点击登录 | 系统锁定账户 15 分钟，提示"密码错误次数过多，请 15 分钟后重试" |
| AC6 | Token 过期 | 用户长时间未操作，Token 已过期 | 发起任意需要认证的请求 | 系统返回 401，前端跳转至登录页 |

## 第五部分：技术规约（独立章节）

> 本章节将技术实现细节与需求描述分离，便于需求评审时聚焦业务价值。

### 5.1 认证与安全

| 维度 | 约定 |
|------|------|
| 用户认证 | JWT Token，access_token 有效期 2h，refresh_token 有效期 7d |
| 密码存储 | bcrypt 加盐哈希，禁止明文存储 |
| 邮箱激活 | 注册后发送唯一激活链接，链接有效期 24h |
| 登录保护 | 连续 5 次失败锁定 15 分钟 |

### 5.2 地图服务

| 维度 | 约定 |
|------|------|
| SDK 选型 | 微信小程序使用腾讯地图（原生支持），Web 端可用高德/百度 |
| 地图选点 | 用户点击地图获取经纬度 + 逆地理编码地址 |
| 附近搜索 | 基于用户当前位置 + 活动坐标计算距离排序 |

### 5.3 AI 审核

| 维度 | 约定 |
|------|------|
| 调用方式 | 异步调用，不阻塞用户提交 |
| 超时策略 | 10s 超时自动降级为人工审核 |
| 兜底机制 | 人工审核接口始终保持可用 |

### 5.4 数据一致性

| 维度 | 约定 |
|------|------|
| 报名防超卖 | 数据库行锁（`SELECT ... FOR UPDATE`） |
| 取消报名名额释放 | 事务保证报名取消与名额恢复原子性 |

### 5.5 分页规范

| 维度 | 约定 |
|------|------|
| 分页方式 | 游标分页（基于时间戳或 ID），避免 offset 深分页问题 |
| 每页条数 | 20 条 |

### 5.6 技术栈建议

| 端 | 技术 |
|----|------|
| 微信小程序 | 原生 / Taro / uni-app |
| Web 管理后台 | React + Ant Design / Vue + Element |
| 后端 | Node.js (Express/Nest) / Python (FastAPI) / Java (Spring Boot) |
| 数据库 | PostgreSQL（推荐，因 API/数据库设计依赖 JSONB 与 PostGIS） |
| 缓存 | Redis（可选，迭代 1 可不引入） |
| 文件存储 | OSS / 本地存储（图片、营业执照等） |
